   .on()?.click()?.bind()?.delegate()?什么？还有.live()…？可能吧，但或许你听都没听说过，为什么有那么多啊，各个API之间有什么区别，实际项目中我究竟该用哪个呢？额，抱歉，我也不知道，讲道理所有的事件绑定都是一个.on()就搞定了？

  当然，理论上来说，其他的都不管，只要记住一个.on()是绑定，.off()是取消绑定就可以了，其他的API其底层写法也都是调用这两个来实现的，只是参数不同而已。那或许文章到这里就可以结束了，没必要再写下去了？每当这个时候我们就会抛出一个问题“那你知道XXX的核心原理和具体实现方式嘛？”的问题，然后回答道“我只管用就行了，怎么实现的who care”。好像一切都是那么顺理成章，我也没有什么特别强悍能够说服你的理由，只是个人觉得，人都已经笨成这样了，天赋不够，只有勤能补拙是良训，一分辛苦一分才来安慰一下自己了。或许这也忒被动了，但是稍作了解，你会发现jquery的事件处理机制是多么的巧妙、精细，特别是拜读了大神艾伦的解密jquery事件核心一文，大有醍醐灌顶、豁然开朗之势。

  多说无益，先从零基础的API开始说起吧。

jquery常用事件绑定处理器

.on()

  .on( events ,[selector ] ,[ data ], handler(eventObject))

  这个算是最常用的了，第一个参数必填——标准事件名称，允许多绑定多个事件，以空格来分割；第二个参数选填，jquery的DOM选择，例如’.class-a’；第三个参数选填，传入到事件里面的数据，通过回调的event.data来获取；第四个参数必填，事件的处理函数。没什么好说的，第二个参数不填就代表是直接处理，填了就代表是委托事件处理什么的都不说了（后面再说），补充两点无关紧要的，如果只想要传入data参数，那么第二个参数填写false就可以了；事件是可以有自己的命名空间的，比如你传入的事件名称是click.me，这个‘.me’就是这个事件的命名空间，如果想要取消绑定这个事件，直接调用.off(‘click.me’)就行了，其他的想到了再补充。

.bind()

  .bind( eventType [eventData ], handler(eventObject))

  为一个指定的已经存在的元素绑定一个或者多个事件。第一个参数是标准事件名称，如果是多个，以空格分隔；第二个参数选填，是绑定的数据（这里吐槽一下，发现在zepto-1.1.6中，参数是一个纯字符串的话是没法触发回调的，如果是{a:’a’}这样的Object的话就可以？jquery倒是正常的，难道是BUG？），在回调函数的参数中获取（.data）；第三个参数是回调函数处理。这里可以补充一下，有两种绑定的写法：第一种是直接.on()那样直接写，如下：

  .one( events [data ], handler(eventObject))

  为一个给定的元素（已经存在，或者将来会存在，将来会存在的意思就是暂时不存在，但是会以.appendTo()等其他方式添加到DOM节点）绑定一个或者多个事件，具体参数和.bind()没有区别，唯一的区别是，这些事件只会生效一次就自动取消了，虽然本质上讲和调用.unbind()没啥区别，不过工具嘛，总是本着大道至简的道理来设计的，如下：

事件处理的由来

  当然，说“由来”这种事，对于我这种小菜鸟来说特别有装逼的嫌疑，动辄大谈历史上下五千年巴拉巴拉。。。不好意思，目前只能据我自己所了解的来说。

  虽然我们都知道，jquery这些所有的事件绑定处理器归根到底都是element.addEventListener(‘event’,handler)来实现的，那我们干嘛不直接这样用呢？可以是可以，不过试想一下，如果有100个dom节点要绑定怎么办呢？如果这个节点是动态生成的呢？如果想取消某一个节点的某一个事件呢？问题总是会随着业务的发展变得越来越多，想必再用原生js来写就比较麻烦了吧，所以用jquery，它实现了这些事件绑定的封装，如上，基本上能够满足大部分的需求了。那么问题来了，这么多处理机制，它究竟是如何处理的如此的有条不紊，使我们用起来的时候觉得非常方便的呢？

DOM事件流

  首先，我们需要了解一下DOM的事件处理模型，简单地说就三个阶段，事件捕获、获取事件目标、事件冒泡。这里借个图：

  标准的DOM事件都是有这三个阶段的（当然极少数不会冒泡），以click事件为例，假设现在我的页面就只有一个button，我给它绑定了一个click事件，那么整个事件的流程就是这样的：

  浏览器会首先从document查找，然后再到body层，然后再往body层往下查找，直到找到绑定事件的那个元素为止，这个查找的过程就是事件捕获阶段；现在绑定事件的目标节点找到了，就是这个button，此时讲道理的话顺理成章的我们可以开始执行这个事件处理的函数了，因为目标已经找到了，直接执行就可以了，这个阶段就是获取事件目标阶段；最后这个事件会沿着之前查找的方向从里到外开始“冒泡”，比如我们刚刚绑定的button，会先往上找到body，然后再往上找到document，不作处理的话，会直到冒泡到document为止，这个阶段就是事件的冒泡。

  有人在想，如果我不想在获取到目标节点的时候执行回调，而是在冒泡阶段，冒泡到某个指定节点才执行回调可以吗？听上去理论上可行的，这个就是常说的“事件委托”。

事件委托

  何为委托呢？就是说这件事本身应该由你来办理，但是出于某种情况的考虑，你不想办理这件事情了，你让别人来帮你做，这就叫委托了。同样的道理，上面我们提到的场景，一个页面里面就一个button，现在我不在获取目标事件的时候就执行回调，而是根据事件的冒泡原理，在冒泡到某一个指定节点比如（body层）的时候再执行。原生的js可没有提供这样的接口，不过jquery帮我们封装好了，比如.on()方法，我们可以这样写：

  这样我们就可以直接把事件委托给body层处理了。为什么要这样做？额，很容易理解吧，比如我们现在body里面有1000个button，每个button都要绑定一个事件，如果不用委托的话，我们可以直接选择button元素然后绑定事件，注意，实际上这里已经绑定了1000个事件。但是如果我们用事件委托来处理的话，效果就不一样了，我们只在body上面绑了一个事件而已。两种方式的内存开销想必是可想而知的吧。再容易理解一点的话，就是现在公司有100个人，都有需要快递签收，是这100个人排队挨个去签收效率高呢还是统一委托给前台小妹一个人效率高呢。

  大致了解了事件委托机制的话，对jquery的一些事件绑定机制就不难理解了，比如.live()方法，是把所有的事件都委托给document对象来执行的，可想而知，事件冒泡的时间是不是太长了，因为要冒泡到document，而且某些事件（比如submit）还不支持冒泡，关键是，一个取消document的绑定，所有的绑定就都失效了，问题可能还不止这些，毕竟把所有的事情都交给一个人来做，如果简单的话还好，一旦复杂了，难免会出各种各样的问题，以至于后来这个api直接被砍掉了。

  jquery的事件绑定都是通过调用.on()来实现的，如果想进一步深入了解，入读.on()的源码即可，将来某个时间想起来了再写写吧。
